# Phase 1: 기획 분석 (페이지별 병렬 분석)

> **핵심**: 첨부된 노션 페이지 하나당 하나의 분석 팀원을 생성하여 병렬 분석한다. 리드는 결과를 종합한다.

> ⚠️ VERIFY: 이 Phase에서 코드를 작성하지 않는다. 분석만.

---

## 0단계: 스냅샷 기준선 자동 비교 (Phase 1 사전 게이트)

> **조건**: `_devnotion_snapshots/_index.md`가 존재할 때만 실행. 없으면 스킵 + 경고 출력.

스냅샷이 존재하면, 분석 시작 전에 **기획서 변경/충돌을 자동 탐지**한다.

### 0-1: 대상 페이지 fetch + 기존 MD와 diff

1. 대상 노션 페이지를 fetch
2. `_index.md`에서 해당 페이지가 속한 기능 폴더를 식별
3. 기존 `{기능}/{문서}.md`를 `_prev.md`로 백업 → 새 fetch를 덮어쓰기
4. `diff -u _prev.md {문서}.md` 실행

### 0-2: 변경 스펙 → 크로스 레퍼런스 비교

1. diff에서 변경된 내용 중 숫자/enum/필드 변경을 추출
2. 해당 기능의 `_summary.md` 갱신 (이전 값 → 현재 값)
3. `_index.md`의 크로스 레퍼런스에서 같은 개념을 참조하는 **다른 기능**의 `_summary.md`와 대조
4. 불일치 발견 시 충돌 후보로 기록

### 0-3: 기획↔코드 불일치 체크

변경된 스펙에 대해 Grep으로 코드 값 검색 → 불일치 표시

### 0-4: 게이트 판정

| 결과 | 동작 |
|------|------|
| 변경 없음 + 충돌 없음 | `✓ 기획서 기준선 일치 확인` → 1단계 진행 |
| 변경 있음, 충돌 없음 | 변경점 요약 출력 → "이대로 진행?" 확인 → 1단계 |
| 충돌 발견 | 충돌 보고서 (`[불일치]`/`[의심]` 태그) → AskUserQuestion 해소 → 1단계 |
| 스냅샷 없음 | `⚠ 기준선 스냅샷 없음. /Dev_notion check init 권장` → 1단계 (스킵) |

> 상세 절차는 `phase_conflict_check.md`의 "모드 3" 참조.

## 1단계: 노션 URL 파싱 및 페이지 목록 확정

1. 사용자 입력에서 노션 URL/검색어를 **모두** 추출한다
2. `ToolSearch`로 노션 도구를 로드: `query: "+notion fetch search"`
3. URL인 경우: 각 URL을 페이지 목록에 추가
4. 검색어인 경우: `mcp__claude_ai_Notion__notion-search`로 검색 → 결과 페이지들을 목록에 추가
5. 최종 페이지 목록을 확정하고, 각 페이지에 인덱스 부여 (page-1, page-2, ...)

---

## 2단계: 분석 팀원 동적 생성 + 병렬 분석 시작

페이지 수(N)에 따라 분기:

**N = 1인 경우:**
- 리드가 직접 분석하거나, 복잡도가 높으면 분석가 1명 spawn
- 3단계(단일 페이지 분석)로 진행

**N ≥ 2인 경우:**
- `page-analyzer-1` ~ `page-analyzer-N` 을 **동시에 spawn** (Agent 도구 병렬 호출)
- 각 분석가에게 전달하는 프롬프트:

```
## 담당: 페이지 분석가 {N}
## 노션 페이지: {URL 또는 페이지 ID}

### 임무
1. 담당 노션 페이지를 fetch하여 내용을 파악하라
2. 하위문서가 있으면 트리아지(핵심/참고/스킵) 후 핵심만 재귀 탐색 (최대 깊이 4)
3. 이미지가 있으면 다운로드 후 시각 분석
4. 기획 요구사항을 추출하고 스펙 신뢰도 등급(A/B/C/D) 평가
5. 요구사항 → 기존 코드 매핑 (수정 대상 파일 식별, Grep/Glob 사용)
6. 분석 결과를 `_devnotion_page{N}_분석.md`에 저장

> **Unity MCP 실사 분석은 3-2단계에서 별도 수행한다.** 분석가 프롬프트에 포함하지 않는다.

### 출력 형식 (_devnotion_page{N}_분석.md)
```markdown
# 페이지 분석: {페이지 제목}

## 문서 구조
- 메인: [제목]
  - 하위: [제목] - [핵심/참고/스킵]

## 핵심 요구사항 (스펙 신뢰도 포함)
1. [요구사항] — 신뢰도: A/B/C/D — 비고: ...

## 데이터/에셋 관련 정보
## 기존 코드 매핑 (수정 대상 파일)
- [파일경로]: [관련 요구사항] — [변경 내용 예상]

## Unity 실사 분석
### 관련 씬/프리팹 구조
- [씬/프리팹명]: [오브젝트 계층 요약]
### 기존 컴포넌트 현황
- [오브젝트명]: [부착된 컴포넌트 목록] — [기획 대비 누락/추가 필요 항목]
### UI 현황 (해당 시)
- [UI 요소명]: [현재 상태] — [기획과의 차이]
### 스크린샷 비교
- [현재 스크린샷 경로] vs [기획 이미지] — [차이점]
### 에셋 의존성
- [에셋명]: [참조 관계 요약]
### 실사에서 발견된 이슈
- [이슈 내용] — [심각도]

## 시스템 간 연관관계
## 참조 이미지 목록
## 잠재적 이슈/모순점
```

### 하지 마라
- 다른 분석가의 담당 페이지를 fetch하지 말 것
- 코드를 수정하지 말 것 (분석만)
- Unity MCP로 오브젝트를 생성/수정/삭제하지 말 것 (조사만)
- 구현 계획을 세우지 말 것 (리드가 종합 후 수행)
```

## 3단계: 하위문서 탐색 + 문서 트리아지 (각 분석가가 수행)

각 분석가(또는 N=1일 때 리드)가 담당 페이지에 대해 수행:

1. 페이지 내용에서 하위 페이지 링크/참조를 **전체 목록**으로 수집
2. 각 하위문서의 **제목/캡션**을 기반으로 3등급 분류:

| 등급 | 기준 | 처리 |
|------|------|------|
| **핵심** | 기능 명세, 데이터 구조, UI 설계, 시스템 요구사항 | 전체 fetch + 상세 분석 |
| **참고** | 일정, 우선순위, 배경 설명, 기술 스택 결정 이유 | 제목만 기록, 필요 시 fetch |
| **스킵** | 회의록, 변경이력, 관련 없는 문서, 아카이브 | 무시 |

3. **핵심 문서만** fetch
4. 핵심 하위 페이지에 또 하위문서가 있으면 재귀 탐색 (최대 깊이 4)
5. 참고 문서는 제목만 체크포인트에 기록
6. 트리아지 결과 + **스펙 신뢰도 등급**을 각자의 `_devnotion_page{N}_분석.md`에 저장

## 4단계: 리드 — 전체 종합 (중요!)

모든 분석가의 결과를 수신한 후, 리드가 종합:

1. 각 `_devnotion_page{N}_분석.md` 파일을 Read로 읽음
2. **교차 분석** 수행:
   - 페이지 간 중복/모순되는 요구사항 식별
   - 공통 수정 대상 파일 식별 (충돌 위험)
   - 시스템 간 연관관계 통합
   - 스펙 신뢰도 종합 (가장 낮은 등급 기준)
   - **Unity 실사 결과 통합**: 각 분석가의 씬/컴포넌트 조사 결과를 종합하여 같은 오브젝트를 다른 페이지에서 수정하려는 충돌 식별
   - **기획 vs 현재 상태 갭 분석**: Unity 실사 스크린샷과 기획 이미지의 차이를 종합
3. 종합 결과를 `_devnotion_phase1_기획분석.md`에 저장:

```
## 기획 문서 전체 요약

### 분석 페이지 목록
- Page 1: [제목] — 분석가: page-analyzer-1
- Page 2: [제목] — 분석가: page-analyzer-2
...

### 페이지 간 교차 분석
- 중복 요구사항: [...]
- 모순점: [...]
- 공통 수정 파일: [파일] — Page 1과 Page 3에서 동시 수정 필요 (충돌 주의)

### 핵심 요구사항 종합 (스펙 신뢰도 등급 포함)
- [요구사항 1] (Page 1) — 신뢰도: A
- [요구사항 2] (Page 2) — 신뢰도: B (필드명 누락)
- [요구사항 3] (Page 1, 3) — 신뢰도: C (사용자 확인 필요)

### 데이터/에셋 관련 정보
### 시스템 간 연관관계 (통합)
### 참조 이미지 목록
```

> ⚠️ CHECKPOINT: 신뢰도 C/D 항목이 있으면 사용자에게 반드시 질문한다.

## 4.5단계: 기획서 충돌 & 기획-구현 모순 분석 (중요!)

> **목적**: 기획서 간의 충돌, 그리고 기획서와 현재 구현 사이의 모순을 **구현 전에** 체계적으로 발견하여 질문한다. 이 단계를 거치지 않으면 구현 중 모순이 발견되어 재작업이 발생한다.

### A. 기획서 간 충돌 탐지 (N ≥ 2일 때)

페이지가 2개 이상이면, 종합 결과(`_devnotion_phase1_기획분석.md`)를 기반으로 아래 관점에서 문서 간 충돌을 탐색:

| 충돌 유형 | 탐지 방법 | 예시 |
|----------|----------|------|
| **스펙 모순** | 같은 시스템/기능에 대해 다른 동작을 기술 | 문서A: "턴제 3턴" / 문서B: "자유 행동" |
| **데이터 구조 불일치** | 같은 엔티티의 필드 정의가 다름 | 문서A: HP는 int / 문서B: HP는 float ratio |
| **우선순위 모순** | 문서 간 구현 순서/중요도가 상충 | 문서A: "이 기능 먼저" / 문서B: "이 기능이 선행 조건" |
| **용어 불일치** | 같은 개념을 다른 이름으로 지칭 | 문서A: "재료" / 문서B: "식재료" / 코드: "Ingredient" |
| **범위 충돌** | 한 문서에서 삭제/대체하려는 기능을 다른 문서가 전제 | 문서A: "구 인벤토리 폐기" / 문서B: "구 인벤토리에 탭 추가" |

### B. 기획서-구현 모순 탐지 (항상 수행)

각 기획 요구사항에 대해, Phase 1에서 매핑된 기존 코드를 대조하여 아래 관점에서 모순을 탐색:

| 모순 유형 | 탐지 방법 | 예시 |
|----------|----------|------|
| **동작 불일치** | 기획의 기대 동작 vs 코드의 실제 로직 비교 | 기획: "손님 최대 5명" / 코드: `MAX_CUSTOMERS = 3` |
| **삭제/변경 의존성** | 기획이 변경을 요구하는 코드를 다른 시스템이 참조 | 기획: "HP를 ratio로 변경" / 전투UI가 int HP에 의존 |
| **데이터 스키마 갭** | 기획의 데이터 구조 vs D_ 클래스/XML 필드 대조 | 기획: "맛 속성 5종" / D_Ingredient: 4종(Salty/Sweet/Sour/Spicy) |
| **누락 인프라** | 기획이 전제하지만 코드에 없는 시스템/API | 기획: "퀘스트 진행도 표시" / 퀘스트 시스템 자체가 없음 |
| **암묵적 전제 충돌** | 기획이 암묵적으로 가정하는 상태 vs 실제 | 기획: "메인메뉴에서 바로 전투" / 실제: StoreScene 경유 필수 |
| **UI/씬 구조 갭** | 기획 UI 설계 vs Unity 실사 분석 결과 비교 | 기획: "인벤토리 5탭" / 현재 InventoryUI: 3탭 |

### C. 충돌 보고서 작성

발견된 충돌/모순을 `_devnotion_phase1_기획분석.md`의 **"충돌/모순 분석"** 섹션에 기록:

```markdown
### 충돌/모순 분석

#### 기획서 간 충돌
| ID | 유형 | 문서A | 문서B | 내용 | 심각도 |
|----|------|-------|-------|------|--------|
| C-01 | 스펙 모순 | [페이지1] | [페이지2] | [충돌 내용] | Critical/High/Medium |

#### 기획서-구현 모순
| ID | 유형 | 기획 요구사항 | 현재 구현 | 충돌 내용 | 심각도 |
|----|------|-------------|----------|----------|--------|
| I-01 | 동작 불일치 | [기획 내용] | [코드 위치:동작] | [모순 설명] | Critical/High/Medium |
| I-02 | 누락 인프라 | [기획 전제] | 해당 없음 | [없는 시스템 설명] | Critical |
```

### D. 충돌 해소 질문 루프

발견된 충돌이 1개 이상이면 AskUserQuestion 루프 수행:

```
반복 (충돌 보고서의 각 항목, 심각도 높은 순):
  AskUserQuestion:
    - 충돌 내용 + 양쪽 근거 제시
    - options:
      - "문서A 우선" → 문서B의 해당 부분을 무시/조정
      - "문서B 우선" → 문서A의 해당 부분을 무시/조정
      - "현재 구현 유지" → 기획 해당 부분 스킵 (기획-구현 모순 시)
      - "기획대로 변경" → 기존 코드 수정 (기획-구현 모순 시)
      - "둘 다 아님 — 별도 지시" → 사용자 자유 입력
      - "통과 (나중에 결정)" → DEFERRED 처리, Phase 4에서 재확인

  모든 Critical/High 항목 해소 시 루프 종료 (Medium은 DEFERRED 허용)
```

충돌 해소 결과는 Phase 2 전략 수립과 Phase 4 구현에 직접 반영한다.
충돌이 0개이면 이 단계를 "충돌 없음 확인"으로 기록하고 다음 단계로 진행.

## 5단계: 기획 검토 — 최악 시나리오 재귀 질문 (중요!)

**종합된** 전체 내용을 기반으로 최악의 경우의 수를 상정하여 검토합니다.
**특히 페이지 간 교차 분석에서 발견된 모순/충돌을 우선 질문합니다.**

아래 루프를 사용자가 **"통과"를 선택할 때까지 반복**:

```
반복:
  1. 아래 관점에서 최악의 시나리오를 상정하고 문제점을 도출:
     - 구현 가능성, 명확성, 일관성, 누락 정보
     - **페이지 간 모순** (교차 분석 결과), 엣지 케이스, 데이터 무결성
     - 스펙 신뢰도 C/D 항목은 반드시 질문에 포함

  3. AskUserQuestion으로 질문:
     - 반드시 "통과" 옵션 포함

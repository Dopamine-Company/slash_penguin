# 충돌 체크 시스템 (스냅샷 기준선 기반)

> **핵심 아이디어**: 전체 기획서를 **기능별로 분할된 MD**로 `_devnotion_snapshots/`에 보관한다.
> `/Dev_notion` 실행 시 대상 페이지를 fetch → 저장된 MD와 diff → 변경점 감지 + 다른 기획서 MD와 충돌 체크 → 구현 진행.
> `check` 모드로 독립 실행하면 구현 없이 충돌/변경 탐지만 수행.

---

## 스냅샷 폴더 구조

```
_devnotion_snapshots/                    ← .gitignore 등록됨
├── _index.md                            ← 총괄 인덱스 (전체 기획서 목차 + 메타)
├── guest/                               ← 기능 단위 폴더
│   ├── _summary.md                      ← 이 기능의 핵심 스펙 요약 (숫자, enum, 필드 등)
│   ├── guest_system.md                  ← 손님 시스템 기획 원문
│   └── intimacy.md                      ← 호감도 하위 기획
├── combat/
│   ├── _summary.md
│   ├── combat_core.md
│   └── card_system.md
├── store/
│   ├── _summary.md
│   └── store_flow.md
├── inventory/
│   ├── _summary.md
│   └── item_system.md
├── encounter/
│   ├── _summary.md
│   ├── story_encounter.md
│   └── event_encounter.md
├── reward/
│   ├── _summary.md
│   └── reward_system.md
└── ...
```

### 파일 역할

| 파일 | 역할 | 할루시네이션 방지 |
|------|------|-----------------|
| `_index.md` | **총괄 인덱스** — 모든 기능 폴더 목록 + 각 기능의 핵심 스펙 포인터 | AI가 전체 구조를 한눈에 파악, 관련 파일만 읽음 |
| `{기능}/_summary.md` | **기능별 스펙 요약** — 숫자 상한, enum 목록, 필드명/타입, 핵심 동작 규칙 | 기계적 비교 대상. 원문 인용 포함 |
| `{기능}/{문서}.md` | **기획 원문** — 노션 페이지를 정규화한 전체 내용 | diff 대상. 변경 감지용 |

## _index.md 형식

```markdown
# 기획서 스냅샷 인덱스

> 마지막 전체 갱신: {YYYY-MM-DD}

## 기능 목록

| 기능 | 폴더 | 문서 수 | 마지막 체크 | 핵심 스펙 |
|------|------|--------|-----------|----------|
| 손님 시스템 | `guest/` | 2 | 2026-03-17 | 최대 손님 수, 호감도 범위, 무드 enum |
| 전투 | `combat/` | 2 | 2026-03-17 | 턴 구조, HP 타입, 카드 존 규칙 |
| 가게 | `store/` | 1 | 2026-03-17 | 시간대 enum, 씬 전환 흐름 |
| ... | | | | |

## 크로스 레퍼런스 (기능 간 공유 개념)

| 개념 | 관련 기능 | 정의 위치 | 값/타입 |
|------|----------|----------|---------|
| TimeOfDay enum | store, encounter | `store/_summary.md` | Morning/Afternoon/Evening/Night |
| FoodRarity enum | inventory, reward | `inventory/_summary.md` | Normal/Rare/Epic/Dreamy |
| HP | combat, guest(호감도?) | `combat/_summary.md` | int, 0~MaxValue |
| ... | | | |
```

> **크로스 레퍼런스**가 핵심. 같은 개념이 여러 기능에서 참조될 때 정의가 하나로 통일되어 있는지 확인하는 근거가 된다.

## _summary.md 형식 (기능별)

```markdown
# {기능명} — 스펙 요약

> 출처: {노션 페이지 제목} ({URL})
> 마지막 갱신: {YYYY-MM-DD}

## 숫자 스펙
| 항목 | 값 | 출처 원문 |
|------|---|----------|
| 최대 손님 수 | 5 | "한 번에 최대 5명의 손님이 방문" |
| 호감도 최대 | 100 | "호감도는 0~100 범위" |

## enum / 상태 목록
| enum명 | 항목 | 출처 원문 |
|--------|------|----------|
| GuestMood | Happy, Neutral, Angry | "손님 기분: 기쁨/보통/화남" |

## 핵심 필드
| 엔티티 | 필드 | 타입 | 출처 원문 |
|--------|------|------|----------|
| Guest | intimacy | int | "호감도 수치 (정수)" |
| Guest | mood | GuestMood | "현재 기분 상태" |

## 핵심 동작 규칙
| 규칙 | 출처 원문 |
|------|----------|
| 호감도 50 이상이면 특별 인카운터 해금 | "호감도가 50에 도달하면..." |

## 다른 기능과의 접점
| 접점 | 관련 기능 | 내용 |
|------|----------|------|
| 전투 승리 시 호감도 +10 | reward | "전투 보상에 호감도 증가 포함" |
```

> **"출처 원문" 열이 할루시네이션 방지의 핵심.** AI가 스펙을 요약할 때 반드시 기획서 원문을 인용하도록 강제한다.

## 모드 1: 초기 구축 (`/Dev_notion check init [노션 URL...]`)

전체 기획서를 처음으로 스냅샷에 저장하는 절차.

### Step 1: 노션 페이지 전체 수집
```
ToolSearch query: "+notion fetch search"
```
- 제공된 URL 전부 fetch
- 각 페이지의 하위문서도 재귀 탐색 (최대 깊이 3)

### Step 2: 기능별 분류

fetch한 문서들을 **기능 단위**로 분류:

```
분류 기준:
1. 노션 문서의 제목/구조에서 기능 키워드 추출
2. 이미 존재하는 폴더 구조와 매칭
3. 매칭 안 되면 새 폴더 생성 (사용자에게 폴더명 확인)
```

분류가 애매한 문서는 AskUserQuestion으로 확인:
```
"이 문서 '{제목}'을 어느 기능으로 분류할까요?"
options: [기존 폴더 목록] + "새 기능 폴더 생성"
```

### Step 3: MD 저장

각 문서를 정규화하여 저장:

```markdown
---
page_id: {노션 페이지 ID}
title: {페이지 제목}
url: {노션 URL}
fetched_at: {YYYY-MM-DD HH:mm}
feature: {기능 폴더명}
---

# {페이지 제목}

{노션 페이지 본문 — 원본 구조 유지}
```

### Step 4: _summary.md 생성

각 기능 폴더에 대해 `_summary.md`를 생성:
- 저장된 원문 MD에서 숫자, enum, 필드, 규칙을 **원문 인용과 함께** 추출
- AI가 해석하지 않고, 기획서에 명시적으로 적힌 것만 추출
- 명시적이지 않은 것은 `[미명시]` 태그

### Step 5: _index.md 생성

전체 기능 목록 + 크로스 레퍼런스 테이블 작성:
- 같은 개념(enum, 필드, 상수)이 여러 _summary.md에 나타나면 크로스 레퍼런스에 등록
- 값이 다르면 즉시 **충돌 후보**로 표시

### Step 6: 초기 충돌 리포트

크로스 레퍼런스에서 값이 불일치하는 항목을 보고:
```markdown
## 초기 충돌 탐지

| ID | 개념 | 기능A | 값A (원문) | 기능B | 값B (원문) |
|----|------|-------|-----------|-------|-----------|
| C-01 | 최대 손님 | guest | "5명" | store | "3명까지 표시" |
```

## 모드 2: 변경 감지 + 충돌 체크 (`/Dev_notion check [노션 URL...]`)

기존 스냅샷이 있을 때, 특정 페이지의 변경점과 충돌을 탐지.

### Step 1: 대상 페이지 fetch

### Step 2: 기존 MD와 diff
```bash
diff -u "_devnotion_snapshots/{기능}/{문서}_prev.md" "_devnotion_snapshots/{기능}/{문서}.md"
```
- 기존 MD를 `_prev.md`로 백업 → 새 fetch를 `.md`에 덮어쓰기 → diff

### Step 3: 변경 분류 (기계적)

| 분류 | 기준 | 할루시네이션 위험 |
|------|------|-----------------|
| 추가 (`+`) | 새 텍스트 | 낮음 — 원문 그대로 |
| 삭제 (`-`) | 없어진 텍스트 | 낮음 — 원문 그대로 |
| 변경 (`-`→`+`) | 같은 위치 내용 변경 | 낮음 — 양쪽 원문 |

### Step 4: _summary.md 갱신

변경된 내용이 _summary.md의 스펙에 영향을 주는지 확인:
- 숫자 변경 → _summary.md의 숫자 스펙 갱신
- enum 변경 → _summary.md의 enum 목록 갱신
- 갱신 시 이전 값도 기록: `이전: 5 → 현재: 8`

### Step 5: 크로스 레퍼런스 재검증

변경된 _summary.md의 스펙을 _index.md의 크로스 레퍼런스와 대조:
- 같은 개념의 값이 다른 기능의 _summary.md와 불일치하면 **충돌**

### Step 6: 기획↔코드 모순 탐지

변경된 스펙에 대해 코드베이스 검색:
```
_summary.md 변경: "최대 손님 5 → 8"
  → Grep: "MAX_CUSTOMER|maxCustomer|requiredSelectionCount" in *.cs
  → 코드 값과 비교
```

### Step 7: 보고서 출력 + 사용자 확인 루프

(기존 보고서 형식 유지 — 확정/의심 분리)

### Step 8: 스냅샷 갱신 확정

사용자 확인 후 `_prev.md` 삭제, diff 파일 삭제.

## 모드 3: 구현 시 자동 비교 (`/Dev_notion [노션 URL]` — 기존 구현 모드)

**Phase 1 시작 전에 자동으로 실행**되는 경량 충돌 체크.

### 자동 실행 조건
- `_devnotion_snapshots/_index.md`가 존재할 때만 실행
- 존재하지 않으면 스킵 (경고: "기준선 스냅샷이 없습니다. `/Dev_notion check init`으로 초기화 권장")

### 자동 실행 절차

1. 대상 노션 페이지를 fetch
2. `_index.md`에서 해당 페이지가 속한 기능 폴더를 찾음
3. 기존 MD와 diff → 변경점 요약 (5줄 이내)
4. 변경된 스펙이 있으면 _summary.md 기반으로 다른 기능과 충돌 체크
5. 코드와 불일치하는 스펙이 있으면 표시

### 결과 처리

| 결과 | 동작 |
|------|------|
| 변경 없음 + 충돌 없음 | "기획서 변경/충돌 없음 확인" → Phase 1 진행 |
| 변경 있음, 충돌 없음 | 변경점 요약 출력 → "이대로 진행?" → Phase 1 |
| 충돌 발견 | 충돌 보고서 → AskUserQuestion 해소 → Phase 1 |

## 할루시네이션 방지 설계

### 계층적 읽기 (컨텍스트 오염 방지)

```
_index.md (전체 목차 + 크로스 레퍼런스)
  ↓ 관련 기능만 식별
{기능}/_summary.md (숫자, enum, 필드 — 원문 인용 포함)
  ↓ 기계적 비교로 충돌 후보 도출
{기능}/{문서}.md (필요 시에만 원문 확인)
```

- **전체 기획서를 한 번에 읽지 않음** → 컨텍스트 윈도우 오염 방지
- _index.md로 관련 기능만 특정 → _summary.md로 스펙 비교 → 원문은 근거 확인 시에만

### 비교 우선순위

| 순위 | 비교 방식 | 할루시네이션 위험 | 보고 태그 |
|------|----------|-----------------|----------|
| 1 | diff 원문 그대로 | 없음 | `[변경]` |
| 2 | _summary.md 숫자/enum 대조 | 매우 낮음 | `[불일치]` |
| 3 | _summary.md ↔ 코드 Grep 매칭 | 낮음 | `[코드 불일치]` |
| 4 | AI의 동작 로직 해석 | 있음 | `[의심]` |

**원칙**: 순위 1~3으로 해결 가능한 것은 4를 사용하지 않는다.

### _summary.md가 핵심인 이유

- 기획서 원문은 자연어라 AI가 해석하면 오류 가능
- _summary.md는 **정형화된 표**로 스펙을 추출 → 기계적 비교 가능
- 추출 시 반드시 **"출처 원문"**을 포함 → 추출 자체의 정확성을 사용자가 검증 가능
- _summary.md가 잘못되면 사용자가 직접 수정 가능 (사람이 관리하는 기준선)

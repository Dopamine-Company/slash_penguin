## 핵심 원칙

### 1. 기획 명세서 우선 (Spec-First)
- 기존 코드와 노션 기획 명세서가 충돌하면 **기획 명세서를 우선**한다.
- 방향이나 내용이 겹치는 경우 기획의 내용으로 대체한다.
- 기존 코드의 동작을 보존하려고 기획을 왜곡하지 않는다.

### 2. 레거시 자동 정리 (안전 삭제)
- 기획 명세서에 의해 대체되는 레거시 코드는 **참조 확인 후** 삭제/리팩토링한다.
- 삭제 전 Grep으로 참조 카운트를 확인한다. **2개 이상 참조가 있으면 AskUserQuestion으로 사용자 확인** 후 삭제한다.
- 참조가 0~1개(자기 자신만)인 경우에만 자동 삭제를 허용한다.
- `// removed`, `_unused` 같은 흔적을 남기지 않고 깨끗하게 정리한다.

### 3. 컨텍스트 보호 (md 중간 저장)
- 각 Phase 완료 시 분석/구현 결과를 `_devnotion_*.md` 체크포인트에 즉시 저장한다.
- 긴 분석 결과나 노션 문서 내용은 컨텍스트 윈도우에 유지하지 않고 `.md` 파일에 저장 후 참조한다.
- 컨텍스트 압축(compaction) 발생 시에도 체크포인트 파일에서 진행 상황을 복원할 수 있다.

### 4. 구현-QA 자동 순환 (loop=true)
- `loop=true` 인자가 포함되면, Phase 5 완료 후 자동으로 `/Frame_QA`를 호출한다.
- QA에서 발견된 버그를 자동으로 수정하고, 다시 QA를 실행하는 순환을 반복한다.
- 순환은 **QA 전체 PASS** 또는 **최대 3회 순환**까지 반복한다.
- 기본값은 `loop=false` (Phase 6에서 보고 후 종료).

## 실행 모드

| 모드 | 인자 | 동작 |
|------|------|------|
| **구현 모드** (기본) | `[노션 URL]` | 기존 Phase 0~6 실행. 스냅샷 존재 시 **Phase 1 전에 자동 diff** |
| **초기 구축** | `check init [노션 URL...]` | 전체 기획서를 기능별 MD로 분할 저장 (초회 1회) |
| **충돌 체크** | `check [노션 URL 또는 검색어...]` | 기획서 변경 감지 + 충돌/모순 탐지만 (구현 없음) |

### 스냅샷 기준선 시스템

전체 기획서를 **기능별로 분할된 MD**로 `_devnotion_snapshots/`에 보관한다.
- `_index.md`: 총괄 인덱스 (기능 목차 + 크로스 레퍼런스)
- `{기능}/_summary.md`: 기능별 핵심 스펙 (숫자, enum, 필드 — **원문 인용 필수**)
- `{기능}/{문서}.md`: 기획 원문 (diff 대상)
- 상세: `phase_conflict_check.md` 참조
- `.gitignore` 등록됨

### 구현 모드에서의 자동 비교

스냅샷(`_index.md`)이 존재하면, Phase 1 진입 전에 자동으로:
1. 대상 페이지 fetch → 기존 MD와 `diff -u`
2. 변경된 스펙을 다른 기능의 `_summary.md`와 교차 비교
3. 코드와 불일치하는 스펙 표시
4. 충돌 발견 시 AskUserQuestion으로 해소 후 Phase 1 진행

## 사전 조건 (Phase 0: 환경 사전 체크)

Phase 진행 전에 수행한다. 실패 항목은 즉시 사용자에게 알리고 대안을 제시한다.

```
1. 노션 API 연결 확인: notion-search 테스트 호출 → 실패 시 "노션 MCP 연결 실패" 안내
2. Agent Teams 환경변수 확인: CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
   → 미설정 시 안내 후 중단
3. (Phase 5 대비) Unity CLI 연결 확인: node .mcp-server/unity-cli.mjs get_state
   → 실패 시 "Phase 5 Unity 세팅은 수동 필요" 경고 (Phase 1~4는 진행 가능)
4. (Phase 4 대비) IDE 진단 도구 확인: IDE diagnostics 도구 존재 여부
   → 미확인 시 "컴파일 에러 확인은 수동 필요" 경고
```

Critical 실패(노션 API, Agent Teams)는 중단, 나머지는 경고 후 계속 진행.

### 외부 서비스 실패 재시도 규칙

| 서비스 | 재시도 | 폴백 |
|--------|--------|------|
| 노션 API fetch | 1회 재시도 (5초 대기) | 실패 시 `_devnotion_snapshots/`의 캐시된 스냅샷 사용. 스냅샷도 없으면 사용자에게 보고 후 중단 |
| 노션 이미지 다운로드 | 1회 재시도 | 실패 시 `[이미지 로드 실패: {URL}]` 태그로 기록, 분석 계속 진행 |
| Unity CLI (WebSocket) | `/reconnect_unity_cli` 안내 | Phase 1~4는 CLI 없이 진행 가능. Phase 5만 차단 |
| 노션 API 쓰기 (Phase 6) | 1회 재시도 (5초 대기) | 실패 시 `_devnotion_result.md`에 로컬 저장 후 사용자에게 수동 기록 안내 |
| IDE diagnostics | 재시도 없음 | `dotnet build` 또는 수동 확인 안내 |

## 팀 구성 (동적 페이지 분석가 + 리드)

> **핵심 변경**: 첨부된 노션 페이지 수(N)에 따라 **페이지당 1명의 분석 팀원**을 동적 생성한다. 각 분석가가 독립적으로 담당 페이지를 분석하고, 리드가 전체 결과를 종합하여 계획을 세운다.

### 고정 역할

| 역할 | 에이전트 이름 | Phase 1 | Phase 2-2.5 | Phase 3 | Phase 4-5 | Phase 6 |
|------|-------------|---------|-------------|---------|-----------|---------|
| **리드 (메인 세션)** | (main) | 노션 URL 파싱 + 분석가 spawn + 결과 종합 | 전략 수립 + AskUserQuestion 루프 | 플랜 종합 + ExitPlanMode | 태스크 분배 + 노션 기입 | 보고 + 팀 정리 |
| **Validator** | validator | 대기 | 대기 | 검증 체크리스트 준비 | 컴파일/정적 분석, [PASS]/[FAIL] | Unity 스크린샷 검증 |

### 동적 역할 — 페이지 분석가 (Phase 1에서 생성)

| 역할 | 에이전트 이름 | Phase 1 | Phase 2-2.5 | Phase 4-5 |
|------|-------------|---------|-------------|-----------|
| **페이지 분석가 1** | page-analyzer-1 | 담당 페이지 fetch + 하위문서 탐색 + 코드 매핑 | **RT 증거 수집** (해당 페이지 범위) | **Builder** — 모듈 구현 |
| **페이지 분석가 2** | page-analyzer-2 | 담당 페이지 fetch + 하위문서 탐색 + 코드 매핑 | **RT 증거 수집** (해당 페이지 범위) | **Builder** — 모듈 구현 |
| **...** | page-analyzer-N | ... | ... | ... |

- 전원 `subagent_type: general-purpose` (Phase 4에서 파일 편집 필요)
- 페이지 분석가: `plan_mode_required: true` (분석 방향 리드 사전 검토)
- Validator: `plan_mode_required: false` (프로토콜이 정형화)

### 팀 규모 결정 규칙

| 조건 | 분석가 수 | 비고 |
|------|----------|------|
| 노션 페이지 1개 | 분석가 1명 | 리드가 직접 분석해도 됨 (판단에 따라) |
| 노션 페이지 2~5개 | 페이지당 1명 | 각 분석가가 병렬 분석 |
| 노션 페이지 6개+ | 페이지당 1명 (최대 6명) | 초과분은 한 분석가에게 2개 배분 |

### 동적 팀 생성 흐름

> **핵심**: 리드가 먼저 노션 URL을 파싱하고, 페이지 수에 따라 분석가를 즉시 spawn한다.

1. **Phase 1 시작**: 리드가 입력에서 노션 URL/검색어를 파싱하여 대상 페이지 목록을 확정한다
2. **페이지 수 = 1**: 리드가 직접 분석하거나, 분석가 1명 spawn (복잡도에 따라 판단)
3. **페이지 수 ≥ 2**: 페이지당 1명의 `page-analyzer-N` spawn → **전원 동시에 병렬 분석 시작**
4. **Phase 3 시작**: Validator spawn
5. **Phase 4**: 분석가들이 Builder로 전환하여 각자 담당 모듈 구현

### delegate 모드

- 리드는 직접 코드 구현/Unity 세팅을 하지 않고 **조율에 집중**한다.
- **예외**: 노션 API 호출, 최악 시나리오 질문 루프, ExitPlanMode는 리드가 직접 수행

### 장애 복구

- 2분 무응답 → 리드가 태스크 재할당 또는 직접 처리
- Builder는 `TaskUpdate description`에 중간 결과 기록
- Validator 장애 시 → 리드가 컴파일 루프로 대체

## 스펙 신뢰도 평가 (cross-verify 적용)

노션 문서 fetch 후, 각 섹션의 스펙 완성도를 평가한다. 스펙이 불충분한 부분을 추측으로 구현하는 것을 방지.

| 등급 | 기준 | 행동 |
|------|------|------|
| **A** | 필드명, 타입, 로직 모두 명시 | 바로 구현 |
| **B** | 대략적 설명만 (필드 누락 등) | 기존 코드 패턴에서 유추 후 구현 |
| **C** | "나중에 추가" / 빈 섹션 | AskUserQuestion으로 사용자에게 확인 |
| **D** | 스펙 자체가 없음 | 구현 불가, 스킵 후 사용자에게 보고 |

Phase 1 체크포인트에 각 요구사항의 신뢰도 등급을 함께 기록한다.

## 의존성 분류 (cross-verify 적용)

Phase 2에서 구현 단위 간 의존성을 3단계로 분류하여, 팀원 병렬 배분의 근거로 사용한다.

| 유형 | 설명 | 팀 배분 |
|------|------|---------|
| **완전 독립** | 서로 무관한 모듈 | Builder A/B에 각각 배분, 동시 구현 |
| **반독립** | A 완료 후 B 시작, C는 독립 | A+C 동시 → A 완료 후 B |
| **완전 순차** | A→B→C 순서 필수 | 하나의 Builder가 순서대로 |

Phase 2 체크포인트의 의존성 그래프에 유형을 명시한다.

## 로컬 체크포인트 시스템

> **핵심**: 긴 노션 문서 내용이나 분석 결과를 메인 컨텍스트 윈도우에 유지하는 대신, `.md` 파일에 저장하고 참조한다. 팀원 스폰 시 전체 분석 결과를 프롬프트에 복붙하지 않고 파일 경로만 전달한다.

#### 체크포인트 파일 규칙

| Phase | 파일명 | 내용 |
|-------|--------|------|
| Phase 1 (분석가) | `_devnotion_page{N}_분석.md` | 각 페이지별 개별 분석 결과 (분석가가 작성) |
| Phase 1 (종합) | `_devnotion_phase1_기획분석.md` | 전체 종합 + 교차 분석 + 요구사항 목록 + **스펙 신뢰도 등급** |
| Phase 2 | `_devnotion_phase2_전략.md` | 플랜의 플랜 + 의존성 그래프 + **의존성 유형 분류** + 구현 순서 |
| Phase 2.5 | `_devnotion_phase25_redteam.md` | 공격 보고서 + 각 항목 처리 결과 (수용/반박/통과) |
| Phase 3 | `_devnotion_phase3_상세계획.md` | 파일/클래스/메서드 수준 상세 계획 |
| Phase 6 | `_devnotion_result.md` | 최종 결과 종합 (아래 템플릿 참조) |

#### 사용 원칙

1. **Phase 완료 시 즉시 저장**: 해당 Phase의 분석 결과를 체크포인트 파일에 기록
2. **팀원 스폰 시 경로만 전달**: `.md` 파일 경로 목록을 전달 → 팀원이 `Read`로 필요한 부분만 읽음
3. **메인 컨텍스트에는 요약만**: 체크포인트에 전체 내용, 컨텍스트 윈도우에는 핵심 요약만 유지
4. **중복 읽기 방지**: 팀원 스폰 시 기존 체크포인트 경로를 명시 → 이미 분석된 내용을 다시 분석하지 않음
5. **구현 완료 후 선택적 정리**: `_devnotion_result.md`는 보존하고, 나머지 체크포인트는 Phase 6에서 AskUserQuestion으로 사용자 확인 후 삭제한다.

#### 팀원 스폰 시 체크포인트 전달 예시

```
## 기존 체크포인트 (중복 읽기 금지)
- _devnotion_phase1_기획분석.md: 노션 문서 요약 + 요구사항 (이미 완료)
- _devnotion_phase2_전략.md: 구현 전략 + 의존성 그래프 (이미 완료)
→ 위 파일에 포함된 정보는 다시 분석하지 말 것. Read로 필요한 부분만 참조.
```

## 노션 기입 공통 절차

각 Phase 완료 시 아래 절차로 노션에 진행 상황을 기록합니다:

### 기입 방식: 갱신 우선 (Upsert)
1. `ToolSearch`로 노션 도구 로드: `query: "+notion fetch update"`
2. `mcp__notion__notion-fetch`로 대상 페이지의 **현재 내용**을 먼저 가져옵니다
3. 기존 내용에 해당 Phase의 기록이 **이미 존재하는지** 확인합니다
4. **기존 기록이 있는 경우** → `replace_content_range` 명령으로 해당 구간을 갱신
5. **기존 기록이 없는 경우** → `insert_content_after` 명령으로 페이지 끝에 추가

### 기록 형식
```
[Phase N 완료] - {날짜}
- 수행 내용: {이번 Phase에서 한 일 요약}
- 상태: {진행중 / 완료 / 이슈 발생}
- 비고: {특이사항이 있으면 기재}
```

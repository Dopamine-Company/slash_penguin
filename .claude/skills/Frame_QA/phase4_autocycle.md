# Phase 4: Auto-Cycle 반복 (auto=true 전용)

> **auto=false이면 이 Phase를 건너뛴다.** Phase 3 보고서 출력 후 종료.
> **auto=true이면 Phase 3 완료 후 이 Phase로 진입한다.**

## 사이클 반복 절차

```
현재 사이클 N (1~10):

1. Phase 3 완료 — 사이클 N 결과를 _qa/history/cycle_N_[시나리오명].txt에 기록

2. N < 10이면:
   a. 플레이 모드 종료 확인 (이미 Phase 3에서 종료됨)
   b. 시나리오 풀에서 다음 시나리오 선택 (이전 사이클 및 이전 실행과 중복 불가 — Phase 0의 시나리오 선택 알고리즘 참조)
   c. 이전 사이클의 _qa/verify/ 스크립트 성장 내역 반영 (이미 Phase 3에서 완료)
   d. **Phase 0 → Phase 1 → Phase 1.5 → Phase 2 → Phase 3** 반복
      - Phase 0: Mermaid 플로우 재로드 (캐시 활용) + 구조 검증
      - Phase 1: Mermaid 노드 기반 Plan 재수립 (EnterPlanMode/ExitPlanMode 호출 안함)
      - Phase 1.5: 새 시나리오용 임시 데이터 준비
      - Phase 2: 실행
      - Phase 3: 정리 + 사이클별 보고
   e. N += 1, 반복

3. N == 10이면:
   a. 10사이클 종합 보고서 출력 (아래 포맷)
   b. Knowledge Cache 쓰기 (아래 "캐시 쓰기 절차" 참조)
   c. scenario_log.md append
   d. 종료
```

## 사이클별 Phase 1 최적화 (2회차 이후)
- 1회차에서 수행한 병렬 에이전트 분석(1~3단계) 결과를 **재사용**한다.
- 2회차 이후에는 **4단계(검증 포인트 도출)부터 시작**하여 새 시나리오에 맞게 검증 포인트와 실행 스크립트만 재작성한다.
- 이전 사이클에서 발견된 이슈를 새 시나리오의 검증 포인트에 반영한다.

## 캐시 쓰기 절차 (종합 보고서 출력 직후, scenario_log.md append 직전에 1회만 실행)

종합 보고서의 "전체 발견 이슈 (중복 제거)" 테이블을 참조하여:

1. 테이블에서 **Critical 또는 Major** 이슈만 추출한다
2. 해당 이슈가 없으면 이 단계를 건너뛴다
3. 해당 이슈가 있으면 `_qa/cache/known_patterns.md`에 **append**한다
4. 파일이 없으면 헤더와 함께 신규 생성한다:

```markdown
# QA Knowledge Cache

> Phase 0에서 자동 로드. 크로스-시나리오 패턴 재발/유사 결함 우선 탐지용.
> 오탐 패턴은 삭제하거나 "회귀 감시: No"로 변경 가능.

---
```

5. 이슈별 append 포맷:

```markdown
### [날짜] 이슈 제목 (발견 시나리오명)
- 원본: 종합 보고서의 해당 이슈 행 그대로 복사
- 규칙: 다른 시나리오에도 적용 가능하게 한 단계 추상화
- 적용 대상: 이 패턴이 영향 줄 수 있는 다른 시나리오/시스템
- 수정됨: Yes/No
- 회귀 감시: Yes
```

6. "규칙" 필드 작성 기준:
   - 특정 변수명/메서드명 대신 **게임 메카닉 수준**으로 기술
   - 예) ❌ "ApplyBuffs() for문 덮어쓰기" → ✅ "다중 효과 중첩 시 적용 순서 검증"
   - 추상화가 어려우면 원본을 그대로 써도 된다. 빈칸으로 두지 않는다.
   - Minor/Cosmetic 이슈는 캐싱하지 않음 (노이즈 방지)
